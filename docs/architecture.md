# 架构详解

> 本文详述 Archon 的设计理念和决策依据。可跳过，但理解后能让你更好地定制。

## 1. 设计目标（约束即价值）

### 问题定义
默认 Claude Code 工作流在中后期有四大失败模式：

| 痛点 | 根因 |
|------|------|
| UI 丑 | 没视觉反馈、没参照、审美是品味活 |
| UX 差 | Claude 没真的用过，不知道哪里挫败 |
| Bug 雪崩 | 改动范围失控、没测试网、没边界 |
| 测试不可靠 | Mock 太多、只测 happy path、测实现非行为 |

这些都不是工具缺陷，是**工作流缺口**。

### 设计目标
1. **用户不被打扰**：只在需求挖掘和交付两端参与
2. **质量有客观证据**：不凭"Claude 说好了"
3. **职责硬边界**：防止"顺手做"制造新 bug
4. **跨项目复用**：预设库 + 动态生成
5. **自主闭环**：AI 能自己验证自己

## 2. 核心架构：Orchestrator + 子 Agent 团队

### 为什么主 agent 只调度？

传统多 agent 系统里主 agent 经常"兼职执行"——既调度又亲自写代码。这导致：
- 上下文污染：主 agent 的对话里混杂各种细节，判断力下降
- 权限失控：主 agent 有所有权限，任何错误都可能放大
- 审查失效：让它自审等于没审

**只调度不执行**的主 agent：
- 只读项目状态 + 派子 agent + 校验结果
- 上下文轻，判断力强
- 权限最小化：只有 `Agent` 工具，不能直接改文件
- 子 agent 有自己的独立 session，彼此独立

### 为什么子 agent 要硬边界？

Claude 的默认倾向是"帮忙到底"——它看到相关代码有问题就顺手修。这在单任务场景好，但在多 agent 协作里是灾难：
- dev-agent 改了测试 → qa-agent 的验证失效
- qa-agent 改了业务代码 → 测试和实现耦合
- 任何 agent 不小心改了需求文档 → 需求漂移

硬边界通过**文件权限 + 工具权限 + 职责声明**三重约束解决。

### 为什么评审要独立 session？

"让写代码那个 agent 自审" = 问学生"你作业对吗"，答案永远是对的。

独立 session 的评审 agent：
- 看不到"任务要求"、"用户期望"等辩护材料
- 只看产物（截图、录屏、diff）
- 像新接手的人一样挑刺

这是把"自动化验证"从主观判断升级为客观评估的关键。

## 3. 五阶段 Kickoff 协议

### 为什么需要这么复杂？

需求不清是所有痛点的上游。90% 的项目延期/返工源自"以为的"和"实际的"不一致。

传统做法的失败：
- **直接让 Claude 猜**：输出大概率不是用户想要的
- **让用户详细描述**：用户自己都没完全想清楚
- **不断来回确认**：效率低，决策疲劳

### 五阶段的意义

| 阶段 | 目的 | 为什么必要 |
|------|------|-----------|
| 5 问建坐标 | 快速获取核心方向 | 不问完全盲猜；问太多用户疲劳 |
| 竞品研究 | 用视觉参照对齐 | 避免"我说 Linear 风你理解成 Bootstrap 风" |
| 可视化选项页 | 让用户做选择而非描述 | 视觉选择比文字描述易 10× |
| A/B 追问 | 挖隐藏细节 | 二选一比开放式易 5× |
| 需求圣经 | 结构化落库 | 后续所有开发对照此文档 |

### 隐藏需求怎么浮现？

阶段 3 的可视化选项页里，**主动加** 2-3 个用户没提但竞品都有的功能。用户看到会反应：
- "哦对，这个我也要"（显性化）
- "这个我不要"（排除，也是信息）
- "诶，我没想过这个可以这么做"（扩展思考）

这是靠"提示"挖需求，比纯问答强得多。

## 4. 自主闭环 vs 端点参与

### 两种极端

| 模式 | 特点 | 问题 |
|------|------|------|
| **全自动** | AI 从头到尾自己做 | 方向偏了没人拉回；最终交付可能完全错 |
| **全手动** | 每一步都问用户 | 用户成瓶颈；失去 AI 速度优势 |

### 端点参与模型的平衡点

- **端点 1：需求挖掘** — 用户投入 30 分钟，定好坐标
- **中段：自主闭环** — 用户零打扰，AI 全速跑
- **端点 2：一次性交付** — 完整可验收成果，用户集中验收
- **反馈闭环** — 异步收反馈，批量处理

**前提**：中段的"自主"必须有硬质量保证。否则变成"全自动模式"的灾难。

### 中段的质量保证

每个 Gate 必须有客观证据：
- Unit test 绿（机械）
- E2E 录屏过完整旅程（客观录屏）
- UI 截图过 ui-critic 独立审查（独立评审）
- UX 录屏过 ux-critic 独立审查
- code-reviewer 独立审查 diff
- Lighthouse / Bundle size 达标

任何一项红 → 回修复 → 最多 retry 5 次 → 超过才 escalate。

## 5. 预设库 + 动态生成

### 为什么要预设库？

Claude Code 写 agent 提示词每次效果不一致：有时写得很好，有时漏掉边界、漏掉输出格式。预设库是"标准件"：
- 硬边界已写死
- 格式统一
- 经过验证

### 为什么要动态生成？

预设库不可能覆盖所有技术栈（Flutter、Vue、Go、Rust、小程序……）。用到时才按需生成，保持库的精简。

### 回写机制的意义

一个项目用到的 `flutter-dev-agent`，验证有效后回写到全局库。下次另一个 Flutter 项目直接复用，不用重造。

**验证门槛**：成功 2 次且成功率 ≥ 80%。防止污染全局库。

## 6. 反馈闭环

### 为什么不直接收到就做？

收到就做的问题：
- 单条反馈派一轮开发 = 高频打扰 + 效率低
- 相关反馈分散处理 = 重复工作
- 没有优先级 = 用户急的没先做

### 批量 + 优先级的价值

triage-agent 的作用：
- **分类**：bug / 优化 / 新需求（不同处理路径）
- **优先级**：P0/P1/P2
- **关联**：相似反馈合并
- **拒绝**：超出范围的新需求先走 kickoff 评估

### "新需求必须走 kickoff"的理由

需求圣经里的"明确不做的功能"是硬边界。新需求直接做 = 范围蔓延 = 回到 bug 雪崩的老路。

正确做法：
- 反馈被标记为 `needs_kickoff`
- 批量积累后 Orchestrator 触发简化版 kickoff
- 用户决定是否纳入需求圣经
- 纳入后走正常开发流程

## 7. 关键取舍

### 为什么用 Playwright 而不是其它？

- **Playwright MCP 存在**：Claude 能直接调用
- **跨浏览器**：chromium/firefox/webkit 一套 API
- **录屏 + 截图 + trace**：证据充分
- **Lighthouse 集成**：性能评估一把梭

替代方案：Cypress（要 GUI 麻烦）、Selenium（慢）、自研（不值）。

### 为什么默认 shadcn/ui？

- **不是库，是模板**：代码在你项目里，可改
- **Radix primitives**：键盘/无障碍/焦点管理默认好
- **Tailwind 原生**：不额外引 runtime
- **社区最活跃**：问题好搜

替代方案：Mantine（库太大）、Ant Design（设计感重，不灵活）、自己写（灾难）。

### 为什么不默认上 CI/CD？

- 项目早期上 CI/CD 拖慢节奏
- 本地 pre-commit hook 已覆盖 80% 需求
- 真上线前再加 CI 也来得及

所以 CI/CD 作为 optional，不在最小集里。

## 8. 可扩展的部分

### 按项目添加 agent

`.claude/agents/_drafts/` 里放项目专用 agent。典型场景：
- 特定业务域（payment-expert、auth-expert）
- 特定任务类型（migration-agent、data-fix-agent）
- 特定工具（figma-importer-agent）

验证 2 次后回写到 `~/.claude/agents/presets/stacks/` 或保留项目级。

### 按团队定制铁律

CLAUDE.md 的六条铁律是起点，不是终点。根据团队习惯加：
- "所有 API 调用必须走 src/lib/api 而非直接 fetch"
- "所有 feature flag 在 src/config/flags.ts 注册"
- "任何 DB migration 必须同时提供 rollback 脚本"

### 按规模调整门槛

Gate 的阈值（retry=5、Lighthouse≥80、bundle≤200KB）是默认值。
不同项目可以调整：
- 早期原型可放松
- 生产级项目可收紧
- 保存在 `~/.claude/agents/presets/config.json`

---

## 10. 最重要的一条

**这套 kit 不是教条，是起点。**

你可以：
- 先用最小集（CLAUDE.md + UX 清单）看效果
- 发现不合适就改预设库
- 积累新 agent 到本地，用着用着生态就厚了

**最大的敌人不是复杂，是"完美但不用"**。先跑起来，边用边改。
