# UX 清单（每个界面完成前必过）

> 由 `ux-critic` agent 对照录屏逐项检查。
> 不打钩不算完成。

---

## 一、状态反馈（所有数据驱动界面必查）

### Loading 状态
- ☐ 数据加载中有视觉反馈（禁止白屏）
- ☐ 用**骨架屏**（首选）或 spinner
- ☐ 骨架屏结构与实际内容一致（不要一坨灰方块）
- ☐ 加载时间 > 3s 有额外提示（如 "正在加载大量数据..."）

**红灯**：点击或打开后 > 300ms 无任何视觉变化

### Empty 状态
- ☐ 列表为空时不是空白页
- ☐ 有插图或图标（视觉分量）
- ☐ 有一句解释（如"你还没有创建过 X"）
- ☐ 有**主要引导操作**（如"创建第一个 X"按钮）

**红灯**：只显示"暂无数据"且没有下一步

### Error 状态
- ☐ 具体错误原因（不是笼统"error"）
- ☐ 用中文（除非国际化）
- ☐ 有重试按钮或恢复路径
- ☐ 网络错误 vs 业务错误 vs 权限错误 分别对待

**红灯**：`Error: undefined` 或纯英文堆栈

### Success 反馈
- ☐ 操作成功有 toast / 状态变化
- ☐ Toast 文案具体（如"已保存 3 项更改"而非"成功"）
- ☐ Toast 2-3 秒自动消失
- ☐ 可手动关闭（× 按钮）

**红灯**：操作完成没有任何反馈

### Disabled 状态
- ☐ 视觉明显区分（opacity ≤ 0.5 或配色灰阶化）
- ☐ `cursor: not-allowed`
- ☐ 有 tooltip 说明禁用原因
- ☐ 禁用条件变化后自动启用（不用用户刷新）

**红灯**：灰色按钮但点击无反应且不解释

## 二、表单（所有有输入的界面）

### 字段校验
- ☐ 失焦即验（blur 触发）
- ☐ 输入时去除现有错误（不阻挠打字）
- ☐ 错误显示在**字段下方**（不是全局 toast）
- ☐ 错误文案指向具体问题（"邮箱必须含 @" 不是"格式错误"）
- ☐ 必填字段有星号 `*`

### 提交
- ☐ 按钮 loading 状态
- ☐ 提交中禁用按钮防止双提交
- ☐ 失败时按钮恢复可点
- ☐ 成功后清空表单或跳转

### 输入辅助
- ☐ 正确的 `type`（email / tel / number）
- ☐ `autocomplete` 属性（email / current-password 等）
- ☐ `inputmode`（移动端键盘类型）
- ☐ 合理的 `placeholder`（可选，且禁止用 placeholder 代替 label）

**红灯**：全局一条"有错误"不指向字段

---

## 三、键盘可达

### Tab 导航
- ☐ Tab 顺序符合视觉顺序
- ☐ 所有可交互元素可 Tab 到
- ☐ 不需要交互的元素 `tabindex="-1"` 或语义化跳过
- ☐ 无键盘陷阱（Modal 打开后 Tab 循环在 Modal 内）

### 键盘快捷键
- ☐ Enter 提交表单（focus 在表单内时）
- ☐ Esc 关闭 Modal / Dropdown / 命令面板
- ☐ Cmd/Ctrl+K 打开搜索（如果有）
- ☐ 方向键浏览列表（如果是列表为主的界面）

### Focus 可见
- ☐ 所有可交互元素有可见 focus ring
- ☐ Focus ring 在深色/浅色背景下都可见
- ☐ 自定义 focus 时不要去掉默认的 `:focus-visible`

**红灯**：`outline: none` 全局去掉了 focus

---

## 四、移动端（375×667 必测）

### 布局
- ☐ 无不必要横向滚动
- ☐ 字号 ≥ 14px（除辅助信息）
- ☐ 行高 ≥ 1.5

### 触控
- ☐ 点击区域 ≥ 44×44px（iOS HIG / Material 标准）
- ☐ 相邻按钮间距 ≥ 8px（防误点）
- ☐ 主要操作在拇指区（下半屏）

### 键盘
- ☐ 键盘弹起不遮挡当前输入框
- ☐ 键盘弹起不遮挡"提交"按钮
- ☐ `type` 正确（email 出现 @，number 出数字键盘）

### 手势
- ☐ 不与浏览器手势冲突（左滑返回）
- ☐ 下拉刷新可选实现
- ☐ 长列表无横向 swipe 意外触发

**红灯**：登录/注册表单在 iPhone Safari 下无法完成

---

## 五、性能感知

### 首屏
- ☐ LCP < 2.5s（Lighthouse 测）
- ☐ 避免字体闪烁（FOIT/FOUT）
- ☐ 图片懒加载（`loading="lazy"`）
- ☐ 关键 CSS inline（如用 Next.js 自动做）

### 交互响应
- ☐ 点击后 < 100ms 有视觉反馈
- ☐ 表单提交等异步操作立即显示 loading
- ☐ 乐观更新（可选，简单场景用）

### 动画
- ☐ 动画 < 300ms（超过会感觉卡顿）
- ☐ 遵循 `prefers-reduced-motion`（无障碍）
- ☐ 用 transform/opacity（避免 layout thrashing）

---

## 六、边界数据（必测）

> 从 qa-agent 的录屏里看

- ☐ 长文本：标题 256 字符、正文 10KB 不破坏布局
- ☐ 空字符串 / 空格：不崩
- ☐ 特殊字符：emoji `👨‍👩‍👧‍👦` / 引号 `"'` / 换行 / 零宽字符
- ☐ 大列表：1000+ 条渲染不卡（虚拟滚动或分页）
- ☐ 慢网络：Slow 3G 不白屏卡死
- ☐ 离线：有明确离线提示
- ☐ 数字：0 / 负数 / 超大数（亿级） 显示正确
- ☐ 日期：时区、夏令时边界

---

## 七、可访问性（A11y）

- ☐ Lighthouse A11y ≥ 90
- ☐ 所有图片有 `alt`（装饰性可空字符串）
- ☐ 表单字段有 `label`（关联 `for` + `id`）
- ☐ 颜色对比度 WCAG AA（正文 4.5:1，大字 3:1）
- ☐ 不仅用颜色传递信息（如红色错误 + 文字说明）
- ☐ 语义化 HTML（`<button>` 不是 `<div role="button">`）
- ☐ ARIA 属性按需（如 `aria-live` 给 toast）
- ☐ 屏幕阅读器测一次（VoiceOver / NVDA）

---

## 八、细节（加分项）

- ☐ 动画过渡自然
- ☐ 空状态有人情味（如插图、幽默文案）
- ☐ Loading 超过 10s 显示"慢网络不是你的错"提示
- ☐ 表单自动保存（长表单）
- ☐ 撤销功能（删除类操作）

---

## 打钩规则

- ✅ 录屏里能看到证据 → 通过
- ❌ 明显没做 → 不通过
- ⚠️ 不确定 → 要求补录屏

**通过标准**（ux-critic 判决）：
- ≥ 90% 通过且无严重失败 → `pass`
- 70-90% → `needs_work`
- < 70% 或有严重失败 → `reject`

严重失败：阻塞核心流程的问题（不是细节），必须 reject。
